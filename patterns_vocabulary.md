# 设计模式 · 全词汇库 + 场景速查表

## 一、核心洞察：设计模式的底层假设

GoF 23 模式建立在 1994 年的计算模型上：

| 假设 | 1994年 | 2024年 |
|------|--------|--------|
| 进程模型 | 单进程 | 多进程/微服务/容器 |
| 内存模型 | 共享内存 | 分布式内存/网络通信 |
| 调用方式 | 同步调用 | 异步/消息/事件驱动 |
| 失败模型 | 确定性失败（抛异常） | 超时、网络分区、级联故障 |
| 状态管理 | 单机状态 | 分布式状态、最终一致性 |

**所以需要的词汇也要扩展：经典模式 + 分布式模式 + 并发模式 + 架构模式。**

## 二、共同词汇全景图（按领域分类）

### 2.1 经典词汇：GoF 23（单进程内）

#### 创建型（Creational）— 对象怎么来
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Singleton** | 全局唯一实例 | "整个系统只能有一个..." |
| **Factory Method** | 子类决定创建什么 | "不知道要创建哪种具体类" |
| **Abstract Factory** | 一套产品族 | "需要配套的对象组合" |
| **Builder** | 分步骤组装 | "对象有很多属性要配置" |
| **Prototype** | 克隆复制 | "基于现有对象改一改" |

#### 结构型（Structural）— 零件怎么拼
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Adapter** | 接口转换兼容 | "旧接口不符合新系统" |
| **Decorator** | 层层包装加功能 | "要给对象加新功能不改原类" |
| **Proxy** | 中间人控制访问 | "访问前要做点事（权限/缓存/日志）" |
| **Facade** | 统一简化入口 | "子系统太复杂，要简化调用" |
| **Bridge** | 抽象和实现分离 | "品牌和实现要能独立变化" |
| **Composite** | 统一处理树结构 | "要处理单个对象和组合对象一样" |
| **Flyweight** | 共享对象省内存 | "大量对象有相同部分" |

#### 行为型（Behavioral）— 对象怎么交互
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Strategy** | 封装算法可替换 | "有多种算法，运行时要切换" |
| **Observer** | 订阅-通知 | "一个变化要通知多个对象" |
| **Template Method** | 固定流程，差异子类填 | "流程一样，某些步骤不同" |
| **Command** | 请求封装成对象 | "要支持撤销、队列、日志" |
| **State** | 状态变行为变 | "对象状态不同，行为不同" |
| **Chain of Responsibility** | 请求逐级传递 | "多个对象都能处理，按顺序试试" |
| **Mediator** | 中间人统一管理 | "对象之间引用太多太乱" |
| **Iterator** | 统一遍历接口 | "要遍历集合但不暴露内部" |
| **Visitor** | 操作从对象中抽出来 | "要对一组对象做不同操作，不想改对象" |
| **Memento** | 保存状态可恢复 | "要能回到之前的状态" |
| **Interpreter** | 规则表达式解析 | "要解释执行简单的规则语言" |

### 2.2 并发词汇（Concurrency Patterns）— 多线程/多进程

GoF 23 没有涉及并发。实际工程中必备：

| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Thread Pool** | 复用线程，避免频繁创建销毁 | "高并发任务要执行" |
| **Producer-Consumer** | 生产者和消费者解耦 | "任务生产和执行速度不匹配" |
| **Read-Write Lock** | 读并发、写独占 | "读多写少的场景" |
| **Future / Promise** | 异步任务的结果占位符 | "调用了异步操作，要拿到结果" |
| **Barrier** | 等待所有线程到达某点 | "多线程分阶段执行，要同步" |
| **Semaphore** | 限制同时访问的资源数 | "资源有限，要控制并发数" |
| **Actor** | 每个对象有自己的线程/邮箱 | "高并发、无共享状态" |
| **Fork-Join** | 分而治之，分任务再合并结果 | "大任务可以拆成独立子任务" |
| **Lock-Free / CAS** | 不用锁，用原子操作 | "锁性能不够，要避免阻塞" |

### 2.3 分布式词汇（Distributed Patterns）— 网络、服务、容错

GoF 23 完全不覆盖。现代架构必备：

#### 服务间通信
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Circuit Breaker** | 服务故障时快速失败，防止级联 | "调用外部服务可能挂" |
| **Retry with Backoff** | 失败重试，间隔越来越长 | "网络不稳定，可能瞬态失败" |
| **Timeout** | 设置等待上限，避免永远阻塞 | "不知道对方什么时候响应" |
| **Idempotency** | 操作多次结果一样 | "网络可能重发，要防重复执行" |
| **Saga** | 长事务拆成多个本地事务，可补偿 | "跨服务事务，不能简单回滚" |
| **Event Sourcing** | 不存状态，只存事件流 | "需要审计、回放、重建历史" |
| **CQRS** | 读和写用不同模型 | "读和写负载差异大" |
| **Outbox** | 先写数据库，再异步发消息 | "要保证消息和事务一致" |
| **API Gateway** | 统一入口，路由聚合 | "客户端要调用多个服务" |
| **BFF** | 为前端定制的后端服务层 | "前端需要聚合多个接口" |

#### 数据一致性
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Two-Phase Commit** | 两阶段投票决定是否提交 | "需要强一致性的事务" |
| **TCC** | Try-Confirm-Cancel，业务层补偿 | "需要最终一致性的分布式事务" |
| **Saga** | 长事务拆分，失败时反向补偿 | "跨多个服务的事务" |
| **Eventual Consistency** | 最终一致，允许暂时不一致 | "分布式系统，网络有延迟" |
| **Conflict Resolution** | 并发修改冲突检测与解决 | "多节点可能同时改同一数据" |
| **Vector Clock** | 判断事件先后顺序 | "分布式系统要判断因果序" |

#### 容错与弹性
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Bulkhead** | 隔离资源，防止故障扩散 | "一个模块故障不能拖垮整个系统" |
| **Rate Limiting** | 限制请求速率 | "防止流量冲垮服务" |
| **Load Shedding** | 过载时主动丢弃部分请求 | "系统扛不住了，保核心功能" |
| **Graceful Degradation** | 故障时降级提供服务 | "主功能挂了，次要功能还能用" |
| **Failover** | 主服务挂了自动切备服务 | "要求高可用，不能停" |
| **Health Check** | 定期检查服务是否健康 | "要自动发现故障节点" |
| **Leader Election** | 多个节点选一个主节点 | "集群需要协调者" |

#### 缓存
| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Cache-Aside** | 应用自己管理缓存读写 | "缓存作为数据库的补充" |
| **Write-Through** | 写同时写缓存和数据库 | "缓存和数据库必须一致" |
| **Write-Behind** | 先写缓存，异步批量写数据库 | "写性能要求高" |
| **Read-Through** | 读缓存miss时自动从数据库加载 | "缓存透明，应用无感知" |
| **Cache Invalidation** | 缓存失效策略 | "数据变了，缓存要同步" |

### 2.4 架构词汇（Architectural Patterns）— 系统级组织

| 词汇 | 一句话 | 使用信号 |
|------|--------|----------|
| **Layered** | 分层：表现层→业务层→数据层 | "系统要分职责" |
| **MVC** | Model-View-Controller分离 | "Web应用，数据展示控制分离" |
| **Microservices** | 小服务独立部署 | "团队大，要独立迭代" |
| **Monolith** | 单块应用，所有代码在一起 | "团队小，简单优先" |
| **Event-Driven** | 事件触发，松耦合 | "系统间要异步通信" |
| **Pipe-Filter** | 数据流过一个接一个处理器 | "数据要逐步处理转换" |
| **Hexagonal / Ports-Adapters** | 核心业务隔离，接口可替换 | "业务逻辑不依赖外部框架" |
| **Clean Architecture** | 依赖指向核心，外层依赖内层 | "要长期维护，测试要容易" |
| **Serverless** | 无服务器，按需执行 | "流量波动大，不想管服务器" |
| **Space-Based** | 数据放内存网格，处理节点弹性伸缩 | "超高并发，需要线性扩展" |

## 五、核心结论

> **GoF 23 是单进程内的设计词汇，解决"对象怎么来、怎么拼、怎么交互"。**
> **现代工程需要扩展：并发词汇 + 分布式词汇 + 架构词汇。**
> 
> 用词汇沟通时，先确认上下文：
> - 在讨论**类设计**？→ GoF 23
> - 在讨论**线程安全**？→ 并发模式
> - 在讨论**服务容错**？→ 分布式模式
> - 在讨论**系统分层**？→ 架构模式
